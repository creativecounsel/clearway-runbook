# SLAs

Turnaround times by request type. The point of an SLA isn't speed for its own sake — it's removing the uncertainty that actually slows people down ("I don't know how long this will take, so I'm starting early/nagging/escalating").

| Request type | Turnaround | Route |
|---|---|---|
| Matches a Clearway entry exactly | Immediate — no review needed | Use as-is |
| Minor variation on something previously cleared | 1 business day | Async (Slack/email) — "Fast Lane" |
| New claim, new data use, new jurisdiction, or new partner type | 3–5 business days | Scheduled review — "Full Review" |
| Anything with legal/regulatory novelty (new framework, new product category) | Case-by-case, flagged at intake | Full Review + relevant SME (privacy, IP, etc.) |

## If a deadline is tighter than the SLA allows

Flag this explicitly and as early as possible — don't wait until the SLA is already blown to mention urgency. Include the actual ship date, not just "ASAP." A real deadline with context (e.g., "this is tied to an external commitment we've already made") changes how a request gets prioritized; a vague "soon" does not.
